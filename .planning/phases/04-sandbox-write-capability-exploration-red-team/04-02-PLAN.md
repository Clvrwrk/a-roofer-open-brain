| Edge Function → AccuLynx API | The sandbox key crosses here on every POST/PUT/DELETE; a wrong key would mutate a live client account |
| Edge Function → prod Postgres | Redacted probe/catalog rows written here |
| Deploy pipeline → Supabase | New function code shipped to the hosted edge runtime |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-04-sandbox | Tampering / Elevation of Privilege | index.ts hard-gate handler top | mitigate | `assertSandbox(SANDBOX_SECRET_NAME)` runs before any AccuLynx fetch (Task 1); acceptance grep asserts it precedes the first fetch and that no acculynx-sync fan-out is imported — only the sandbox NAME is ever resolvable |
| T-04-prodfirst | Tampering | dependency walk (all POST/PUT/DELETE) | mitigate | Every mutating call targets the sandbox account exclusively because the only resolvable key is the sandbox secret; no per-account loop exists (grep asserts zero registry references) |
| T-04-pii | Information Disclosure | probe-row persistence | mitigate | Both response `payload_sample` and outbound `request_body_sample` pass through `redactSample` before insert (Task 2 acceptance) |
| T-04-secret | Information Disclosure | index.ts + deploy logs | mitigate | Key resolved only via `Deno.env.get(SANDBOX_SECRET_NAME)`; the resolved value is never logged or returned; error bodies reference the NAME only |
| T-04-dos | Denial of Service (sandbox pollution) | idempotency-retry probes | accept | D-04 tags every created entity with a run_tag so accumulated test data stays identifiable; sandbox is explicitly disposable — accepted risk, not a blocker |
| T-04-SC | Tampering | package installs | accept | No new packages (reuses jsr:@supabase/supabase-js@2 already in prod); no install task |
</threat_model>
