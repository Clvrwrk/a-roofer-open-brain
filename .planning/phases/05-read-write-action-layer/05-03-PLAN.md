<objective>
Wire the human-approval-gated write path through the Command Center by connecting the two systems that have never been joined (RESEARCH central finding): (1) add the D-09 prod-write permission to `access-control.ts`; (2) add a new agent-authored enqueue route + a shared pending-write mapper (`acculynx-pending-write.ts`); (3) add a `loadPendingAccuLynxWriteSurface()` branch to `live-work.ts` so pending writes appear on the dashboard (RQ-1 — the enqueue gap); (4) extend `decision.ts` with an `acculynx-write-action:*` fallback lookup, the D-09 barrier #2 permission check, and the synchronous edge-function invocation on approve (OQ-3); (5) fire the D-08 notify-only Slack message at enqueue time.

Purpose: Turn the isolated executor (Plan 01) and tables (Plan 02) into an end-to-end approval loop: agent enqueues → dashboard shows dry-run preview → human approves with the right permission → edge function executes → audit logged. This is the SC1/SC2/SC4 backbone.
Output: One new permission, one new lib module + test, one new API route, and two extended existing files — all on the contrib branch, deployed to prod via Coolify (Command Center) in Wave 3's proof.
</objective>

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| agent → enqueue route | Agent (bearer/service token) creates a pending write; actor type + permission gate the boundary |
| human approver → decision route | WorkOS-session human approves; permission + department + prod-write checks gate the boundary |
| decision route → edge function | Service-role fetch crosses into the write executor; only on approve |
| enqueue route → Slack | Notify-only; no approval authority crosses here (D-08) |

## STRIDE Threat Register

| Threat ID | Category | Component | Severity | Disposition | Mitigation Plan |
|-----------|----------|-----------|----------|-------------|-----------------|
| T-05-12 | Elevation of Privilege | prod-write approval | high | mitigate | D-09 barrier #2: decision.ts returns 403 for a prod-target item unless `hasPermission(actor, "approval.decide_prod_write")`; the permission is granted only to a named approver roster, not all humans. Independent of the edge function's assertTarget (barrier #1). |
| T-05-13 | Tampering (approval-gate bypass) | edge invocation | high | mitigate | The edge function is invoked ONLY when `decision === "approve"` for an acculynx-write-action item; no other decision path calls it, and the enqueue route cannot execute (it only creates pending rows). SC4. |
| T-05-14 | Spoofing / Elevation | enqueue route | high | mitigate | Enqueue reuses intake.ts's exact gate: agent-actor-type only + `evidence.attach` permission + department scope; no human-only or execute capability is exposed here. |
| T-05-15 | Repudiation | audit trail | medium | mitigate | Every pending write mirrors into dashboard_action_log at enqueue and the execute writes acculynx_write_action_log; the decision itself is recorded by the unmodified recordLiveWorkDecision (SC2). |
| T-05-16 | Information Disclosure | Slack notify | low | accept | Notify-only message links to the dashboard and contains no secret/PII beyond a redacted summary; Slack is not an approval surface (D-08). |
| T-05-17 | Tampering | shared mapper drift | medium | mitigate | A single exported `mapPendingWriteToLiveWorkItem` is used by both live-work.ts and decision.ts — no duplicated mapping that could diverge between the display surface and the decision lookup. |
</threat_model>
