## STRIDE Threat Register

| Threat ID | Category | Component | Severity | Disposition | Mitigation Plan |
|-----------|----------|-----------|----------|-------------|-----------------|
| T-05-18 | Tampering / Elevation | first prod payment | high | mitigate | The prod payment fires only after: sandbox proof (Task 3), roster config (Task 4), explicit prod target + accountKey, assertTarget (barrier #1), and a permitted approver's dashboard click (barrier #2). Checkpoint is blocking-human, never auto-approvable. |
| T-05-19 | Tampering / Repudiation | payment double-fire | high | mitigate | Idempotency key checked pre-execute + DB-unique; Task 3 explicitly re-tests that a repeated idempotency key fires no second write; Task 5 confirms exactly one prod payment. |
| T-05-20 | Elevation of Privilege | prod-target confusion | high | mitigate | assertTarget refuses implicit/ambiguous targets; the dry-run preview shows an unmistakable PROD marker; a non-permitted approver gets 403 (Task 5 step 3). |
| T-05-21 | Tampering | deploy drift (live vs dev) | medium | mitigate | Converge contrib -> main BEFORE deploy; poll /healthz buildCommit to confirm the pushed SHA is live; explain-then-ship before push (deploy contract). |
| T-05-22 | Information Disclosure | audit-row leakage | medium | mitigate | acculynx_write_action_log stores only redacted request/response (redactSample); no AccuLynx key or raw PII persisted (hard rule 2). |
| T-05-23 | Repudiation | prod write without audit | high | mitigate | Every execute writes an acculynx_write_action_log row with approver + target + idempotency key; Task 4 confirms the row before closing (SC2). |
</threat_model>
