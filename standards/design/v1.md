# Standard: Design Work Products — v1

QC Owner: Quality Control agent
Effective date: 2026-06-01
Supersedes: none (initial version)
Auditor reference key: standards/design/v1.md

---

## Scope

This standard governs **every visual asset the brain produces or publishes** — so that all of them follow the same design principles instead of each agent inventing its own colors and fonts. It covers:

- Website page copy and layout drafts that carry visual styling (routed from `@ob-marketing`).
- Property Card renders and any place-memory artifact surfaced to a human.
- Generated graphics, social images, one-pagers, slide decks, and email templates.
- Dashboard and admin-interface views (`docs/13-...`) and the agent app/Slack surfaces (`docs/12-...`).
- Any new component, token, or theme proposed for a client brand.

The contract these assets conform to is the **DESIGN.md format** (Google Labs, vendored at `standards/design/vendor/design.md/spec.md`, Apache-2.0). The **active brand token file** is `config/brand/DESIGN.md`. DESIGN.md tokens are the normative values; the prose is the rationale. When an asset's styling disagrees with the brand DESIGN.md, the DESIGN.md wins.

This standard checks **conformance to the brand's design system**. It does not replace the marketing standard (`standards/marketing/v1.md`), which still governs consent, PII, EEAT, and schema.org. A published web asset must pass **both**.

## Pass/Fail Checks

### Brand token file (`config/brand/DESIGN.md`)

- [ ] **DSN-001:** The file lints with **zero errors** under `npx @google/design.md lint` (run via `scripts/lint-design.sh`). The `broken-ref` rule is the only error-level rule; any unresolved `{token.reference}` is an automatic fail. — REQUIRED
- [ ] **DSN-002:** A `primary` color is defined (no `missing-primary` warning). The brand must not rely on an agent auto-generating its anchor color. — REQUIRED
- [ ] **DSN-003:** Typography tokens are defined (no `missing-typography` warning). Deployed fonts are explicit, never left to a consumer default. — REQUIRED
- [ ] **DSN-004:** Sections appear in the canonical order from the spec — Overview, Colors, Typography, Layout, Elevation & Depth, Shapes, Components, Do's and Don'ts. No `section-order` warning, no duplicate `## Colors`-style heading (which the spec rejects as an error). — REQUIRED
- [ ] **DSN-005:** Every component `backgroundColor`/`textColor` pair used for normal body text meets WCAG AA (≥ 4.5:1). The linter's `contrast-ratio` rule surfaces violations; the brand file documents and justifies any intentional exception (e.g. large-text-only pairings) in the Do's and Don'ts section. — REQUIRED
- [ ] **DSN-006:** `version`, `name`, and `description` front-matter keys are present. — RECOMMENDED
- [ ] **DSN-007:** Orphaned color tokens (defined but referenced by no component) are flagged for review but do not fail — palette colors used only in prose are legitimate. — RECOMMENDED

### Derived assets (anything rendered from the brand)

- [ ] **DSN-010:** Every color, font, radius, and spacing value used in the asset resolves to a token in `config/brand/DESIGN.md`. Hard-coded hex values, off-palette fonts, or one-off radii that bypass the token system fail. — REQUIRED
- [ ] **DSN-011:** The **five-color role discipline** is respected. Each of the five brand colors keeps its single non-negotiable role; roles are not swapped. Specifically: the tertiary/CTA color (flag red) appears on interactive triggers only, never as decoration; the accent color (golden orange) is never used as a button or interactive trigger; the secondary color (hunter green) is not used on commercial-audience-primary surfaces. — REQUIRED
- [ ] **DSN-012:** At most **one primary CTA** (the tertiary/flag-red `button-primary`) per viewport. Two flag-red CTAs in the same visual field fail. — REQUIRED
- [ ] **DSN-013:** Monospace typography (`mono-*` tokens) appears **only** on Property Card / place-memory surfaces. Any other component using mono fails; the request is escalated as a question of whether the surface should become a Property Card, never whether mono should spread. — REQUIRED
- [ ] **DSN-014:** No new CSS custom properties or design values are introduced outside the token system. A value needed repeatedly is promoted to a token in the brand DESIGN.md, not search-and-replaced into the codebase. — REQUIRED
- [ ] **DSN-015:** Radii within a single visual group are uniform — no mixing of the softest (`xl`) and sharpest (`sm`) corner scales in one card group. — RECOMMENDED
- [ ] **DSN-016:** A change to a typeface token is mirrored across every documented font touch point named in the brand file's Do's and Don'ts before the asset ships (partial font updates fail). — CONDITIONAL

### Phase / era discipline

- [ ] **DSN-020:** A migration of display/heading tokens to the Phase 2 typeface (Archivo) is performed **only** after the Phase 2 brand pass is explicitly initiated — the token update in `config/brand/DESIGN.md` is the trigger. An asset that pre-emptively uses Phase 2 fonts while the brand file still specifies Phase 1 (Inter) fails. — CONDITIONAL

## Escalation conditions

- The brand DESIGN.md fails to lint with errors and the producing agent cannot resolve the broken reference from the defined tokens — escalate; do not ship the dependent asset.
- A requested asset cannot be expressed within the existing token set (a genuinely new component or color role is needed) — escalate to QC for a token addition, rather than introducing an off-system value (would otherwise fail DSN-010/DSN-014).
- A surface requests monospace type — escalate the product question (Property Card promotion?) per DSN-013; do not grant mono inline.
- Any proposed swap of a brand color's role (DSN-011) escalates to the AM before the asset proceeds — role discipline is the brand and is not an Auditor-level judgment call.

---

## Notes for agents

- Run the linter before declaring any brand-file change done: `scripts/lint-design.sh config/brand/DESIGN.md`.
- The spec this standard enforces is vendored at `standards/design/vendor/design.md/spec.md` (Apache-2.0, Google Labs). Read it when a token type or section rule is ambiguous.
- Tokens export to Tailwind v3/v4 and W3C DTCG via `npx @google/design.md export` — use that path to generate theme files rather than re-typing values.
