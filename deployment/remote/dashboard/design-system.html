Principles

Four ideas govern every decision in this system. When a new screen is designed, it is measured against these before it ships.

1 · Five-color role discipline (DSN-011)

Each brand color owns exactly one job. A color is never borrowed for decoration. The affirmative CTA is the only flag-red on screen; the accent never becomes a button; hunter green means success and nothing else.

2 · Contrast-Layer depth

Separation comes from borders and surface tints, not drop shadows. Shadows (--shadow-float) are reserved for genuinely floating layers and selection emphasis — the work surface itself stays flat.

3 · One affirmative action per field (DSN-012)

In any viewport there is a single flag-red primary action. Everything else is a navy or ghost secondary. The eye should never have to choose between two "yes" buttons.

4 · On-system or off (DSN-010 / DSN-014)

Every color, radius, and font is a named token. There are no one-off hex values, no ad-hoc pixel radii, no second typeface. If a value is needed and absent, it is added to the token set deliberately — not inlined.

Layout & spacing

Alignment is structural, not eyeballed. Every band on a page — full-bleed or not — hangs off one content rail, and every gap comes from a 4px spacing scale. Magic numbers are how rhythm drifts; tokens are how it holds (DSN-010 / DSN-014, now extended to space).

The container — one rail for every band

Cap the measure, centre it, and make only the gutter fluid. Full-bleed backgrounds live on the band element; their content sits in .container so it lines up with everything else. The masthead, the sticky nav, and the body all share this — there is no second definition of "the column."

Fluid within bounds — why not pure % or pure fixed

The model

  • max-width caps the measure (readability).
  • margin-inline:auto centres it.
  • clamp() gutter gives the "%-feel" — fluid space that grows with the screen.

Why not the extremes

  • Pure % → 2000px+ lines on a wide monitor; unreadable.
  • Pure fixed → cramped on small screens, dead space on large; never scales.
  • App chrome is the exception — fixed sidebars (e.g. the 280px rail) should stay fixed, not rubber-band.

Patterns

Recurring behaviors that combine the primitives above.

Copy-then-act reminder

When a workflow requires copying content before the next step, the next-step button blinks accent ↔ flag-red until clicked, then settles. It is the one sanctioned use of motion.

Two-column work layout

Accessibility

DSN-005: every text/background pairing meets WCAG AA. The notable consequences:

  • Eyebrows shift by surface. Golden-orange eyebrows pass on navy (~6.5:1) but fail on white, so light-surface eyebrows use on-surface-secondary (#374151, 9.3:1).
  • Status pills carry text + fill, never color alone — "Ready" reads as a word, not just a green shape.
  • Focus is visible. Inputs move their border to navy; nav and segmented items expose a :focus-visible state.
  • Interactive targets hold a ≥40px min-height (buttons) / 42px (nav) / 32px (segments) for comfortable hit areas.
  • Motion is minimal and purposeful — the single blink pattern signals a required action, not decoration.

Defensive styling

The masthead once shipped unreadable code chips — a light fill with no declared text color, inheriting white from the navy band. The lesson is structural, not a matter of being more careful next time: make the bug impossible to express, then let tooling catch what still slips through.

The rule

Anything that sets a background must also declare its own color. An element that fills its background but inherits its ink is making a bet about its surroundings — and that bet breaks the moment it moves to a different surface.

Do

  • Declare color on every filled or bordered element.
  • Let surfaces provide ink; let primitives consume it.
  • Extend the existing --color-on-* pairs to every primitive.

Don't

  • Rely on inherited color for anything with a background.
  • Hard-code a per-instance override for one placement (e.g. "fix it on the hero").
  • Assume a component only ever lands on a light surface.

Contextual tokens — surfaces provide, primitives consume

Define foreground/background pairs on each surface context. Primitives reference the contextual variable, so they render correctly anywhere with zero per-instance overrides. This generalizes the --color-on-primary / --color-on-accent convention the palette already uses.

/* Surfaces PROVIDE the ink */
:root,
.surface-light { --fg: #111827; --chip-bg: #f3f4f6; --chip-fg: #11133f; }
.surface-dark  { --fg: #ffffff; --chip-bg: rgba(255,255,255,.14); --chip-fg: #ffffff; }

/* Primitives CONSUME it — placement-agnostic */
code { color: var(--chip-fg); background: var(--chip-bg); }

A <code> styled this way is correct on light or dark with no overrides. The masthead bug could not have occurred.

Astro notes

How this system is meant to live in an Astro codebase — where the contrast bug recurs most, because components are authored in isolation and composed somewhere else entirely.

Where tokens live

Keep the token layer global; scope only component-specific rules.

  • Put the :root tokens and contextual .surface-* pairs in one global stylesheet imported once in the root layout — not in a scoped <style> block, or the custom properties won't reliably reach every component.
  • Astro scopes each component's <style> by default, but inheritance (color, font) and custom properties still cascade through. Scoping does not protect you from the inherited-ink bug.
  • Reach for <style is:global> only for the token layer; keep everything else scoped.

A Surface component sets the context

Wrap any dark (or otherwise non-default) region in a component that switches the contextual variables. Primitives dropped inside it adapt automatically.

--- Surface.astro ---
const { variant = "light" } = Astro.props;
// <div class={`surface-${variant}`}><slot /></div>

--- usage ---
<Surface variant="dark">
  <Masthead />   /* Code, Pill, Card inside all read --chip-fg */
</Surface>

The primitive never knows its surface; the surface declares the ink. That inversion is what makes correctness independent of the prompt.

Enforce on the build output

Astro compiles to static HTML, which is ideal for automated auditing.

  • Run stylelint over .astro and .css files (pre-commit + CI).
  • After astro build, run @axe-core/playwright or pa11y-ci against dist/ and fail the pipeline on any contrast violation.
  • This is the guard that makes the bug impossible to ship, regardless of how any single component was generated.
