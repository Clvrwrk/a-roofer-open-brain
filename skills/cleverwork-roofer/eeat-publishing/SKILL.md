# EEAT Publish

From a consented soft atom to a publication-ready draft — with two consent gates
standing between raw memory and public web content, and a schema.org packet ready
to go alongside it.

---

## Context Required

- Soft atom: content, EEAT signal type (Experience / Expertise / Authoritativeness / Trustworthiness), value score, consent timestamps
- Job record: project type (reroof, storm replacement, repair, new construction), location (city + state), materials (manufacturer, product), approximate project timeline
- Homeowner or client: name (if consented to attribution — check original_practitioner.consent_to_attribute), city and state only by default
- CompanyCam atoms: any photos linked to this job with is_publishable flag
- Brand voice: pulled from `config/roofer.config.yaml` → `marketing.brand_voice` (tone descriptors, audience, key messages)

---

## Process

### Step 1 — Dual Consent Check (Mandatory Gate)

Before generating any content, verify ALL of the following. If any check fails, STOP and return a clear explanation of what is missing.

**Check 1 — Atom consent:**
- `eeat_signal.publishable_with_consent = true` ✓
- `eeat_signal.consent_recorded_at` is set and is not null ✓
- `consent_flags.publishable_external = true` ✓
- `consent_flags.expires_at` is either null or in the future ✓

**Check 2 — Attribution consent:**
- If the draft will name the homeowner: `original_practitioner.consent_to_attribute = true` ✓
- If the draft will name a crew member: same flag on that practitioner atom ✓
- If the draft will mention a street address or specific property: `consent_flags.cross_client_shareable = true` AND explicit property-level consent from the client recorded ✓

**Check 3 — PII check:**
- No phone numbers, email addresses, or financial amounts from the job appear in the proposed draft content ✓

If all checks pass, proceed. If any fail, return a structured explanation listing each failed check and what action the human must take to resolve it.

### Step 2 — Auditor Pre-Check

Route the atom and its consent metadata to Auditor before drafting. Auditor verifies:
- Consent fields are present and correctly set
- No competitive information or pricing data is present in the atom
- No PII is embedded in the atom content

Auditor PASS required before Step 3. Auditor FAIL returns with specific corrections; do not proceed.

### Step 3 — Frame the Story

Based on the atom content and job context, identify:
- The EEAT signal type this atom primarily carries:
  - Experience: the crew did something noteworthy, went above and beyond, a human moment happened
  - Expertise: a technical insight, code knowledge, materials knowledge was demonstrated
  - Authoritativeness: a long-standing relationship, a referral, a repeat client
  - Trustworthiness: a warranty resolution, a problem discovered and disclosed, a promise kept

- The emotional core of the atom (the sentence or phrase that would make a prospective customer trust this contractor)

- The factual anchors from the job record: what was done, where (city + state), with what materials, with what manufacturer warranty

### Step 4 — Draft the Publication

Write in the brand voice from `roofer.config.yaml`. The draft must:

1. Open with the emotional core — not a project description. The reader should feel the point before they understand the scope.
2. Provide factual grounding in the second paragraph: project type, location (city + state), materials, manufacturer, warranty tier.
3. Include a direct quote if the homeowner consented to attribution — use the exact language from the atom, lightly edited for readability (no changes to meaning).
4. Close with a forward-facing statement: what this job demonstrates about the contractor's practice.
5. End with a call to action appropriate to the brand's configured CTA style.

Length targets:
- Testimonial: 100–200 words
- Project story: 250–400 words
- Case study: 400–700 words
- Blog post: 600–900 words

Do not fabricate details. If the atom and job record do not support a specific factual claim, omit the claim.

### Step 5 — Generate Conductor Approval DM

Compose the Slack DM that Conductor will send to the client's designated approver:

```
Marketing prepared a draft [content_format] from your [project_type] job at [city, state].
About [N] words — takes ~2 minutes to review.

[Link to draft preview]

Please choose:
  ✅ Approve — publishes immediately
  ✏️ Edit — reply in thread with changes
  ⏭️ Skip — this one stays in the brain, won't be proposed again
```

### Step 6 — Post-Approval Capture

After human approval:
- Update the soft atom: `consent_flags.publishable_external = true`, `cold_archive_status = live`
- Write a new publication atom with:
  - `trust_tier = instruction`
  - `eeat_signal.consent_recorded_at` confirmed
  - `source_type = cleverwork_output`
  - `property_id`, `job_id`, `client_id`
  - URL of the live published page (added after CMS publish)
- Route the schema_markup_request to the `schema-markup` skill

## Output Format

Publication draft in Markdown. Schema markup input packet in structured JSON format. Conductor DM in plain text.

---

## Judgment Rules

- Consent is binary. Either both checks pass or the skill stops. There is no partial consent.
- Never use a homeowner's name in a publication without explicit attribution consent in the atom — first name only is acceptable if full name consent is absent but first name consent is documented.
- Never mention a property street address in a publication unless `consent_flags.cross_client_shareable = true` AND explicit property-level consent is recorded.
- Do not editorialize about the homeowner's situation (financial distress, age, vulnerability) — these are the most sensitive soft atoms and must be handled with discretion.
- If the atom describes a complaint or a problem the contractor caused, do not publish it without explicit human review and a clear resolution arc in the draft.

## Notes

- EEAT flywheel velocity: the target is one published piece per 3–5 closed jobs. Volume is irrelevant; authenticity is the moat. Better to publish one exceptional project story per month than to publish diluted content weekly.
- The dual consent check (atom consent + Auditor) is not optional even when the human is in a hurry. The consent check is what separates this flywheel from a generic content marketing tool.
- Competitor confidentiality: if the soft atom mentions a competing contractor by name (e.g., "the previous roofer did X wrong"), that passage must be removed or generalized before publishing. The brain captures it; the publication omits it.
